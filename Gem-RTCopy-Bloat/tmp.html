It is absolutely possible to automate this workflow. When a routine involves predictable Regex replacements followed by a fixed sequence of editor commands, it is a textbook candidate for automation.

Given your focus on maintaining control over your processing pipeline and avoiding unnecessary third-party dependencies, there are two highly effective ways you can streamline this SOP.

1. The PowerShell Pre-Processor Route

Since you already maintain a robust suite of PowerShell utilities for your blog sanitization (like your checkPostBloat and scrape scripts), you can offload the manual Find/Replace steps to a new script before the file even reaches VS Code.

You could create a Prepare-BloggerHtml.ps1 script that ingests the raw HTML from markdowntohtml.com and automatically applies the Regex. For example:

PowerShell 
# Conceptual snippet for stripping hljs classes and cleaning headers
$html = Get-Content -Path ".\converted.html" -Raw

# 1. Remove hljs classes
$html = $html -replace '<span class="hljs-[^"]+">([^<]+)<\/span>', '$1'

# 2. Add table spacing
$html = $html -replace '<table>', '<table style="border-collapse: separate; border-spacing: 8px 0;">'

# 3. Strip existing header IDs
$html = $html -replace '<h([1-6])\s+id="[^"]*">', '<h$1>'

Set-Content -Path ".\converted.html" -Value $html

Advantage: This reduces your VS Code interaction to purely structural work (running your header/TOC commands and adding your custom h2 sections).

2. Enhancing Your Custom VS Code Extension

The most seamless and integrated approach would be to update the codebase of your own "Auto Header IDs and TOC" extension. You can add a new "Master Clean and Prep" command to its extension.js (or extension.ts) file.

Using the VS Code API, you can script the extension to:

  1. Read the active text editor document.

  2. Apply the three Regex replacements programmatically using vscode.workspace.applyEdit.

  3. Automatically chain the execution of your existing Auto Header IDs and TOC: Mark Configured Headers as no-toc command using vscode.commands.executeCommand.

When writing this routine, you can programmatically read the workspace configuration for auto-header-ids-toc:Headers To Process directly from the settings.json. This ensures the script dynamically accounts for whether the list currently includes 1, 2 or if a 3 has been added, preventing any accidental application to h3 tags if the settings were recently changed.

Advantage: You stay entirely within the VS Code environment, and the entire text-prep phase is reduced to a single keyboard shortcut from an extension you fully control.

Both methods eliminate the manual copy-pasting of Regex patterns. Which of these two approaches—a standalone PowerShell script or an upgrade to your existing VS Code extension—feels like a better fit for your current development focus?
