Section 2. Providing REST API in PHP

You said:
Hmm. Does PHP provide a way to expose any PHP function as a webservice accesible via SOAP and/or REST API?
ChatGPT said:

Excellent 👌 — this is the key nuance.

🔹 Raw PHP itself

  • No , plain PHP does not natively expose arbitrary functions as web services.

  • PHP is a general-purpose scripting language: you write scripts ( .php files) that process HTTP requests and return HTML/JSON/XML.

  • If you want SOAP or REST APIs , you must either:

    • Write the routing + serialization logic yourself, or

    • Use a library/framework.

🔹 SOAP in PHP

PHP has a built-in SOAP extension ( SoapServer , SoapClient ).

  • Lets you create a SOAP server that maps WSDL operations to PHP functions.

  • Example:

function add($a, $b) { return $a + $b; } $server = new SoapServer(null, ['uri' => "http://localhost/soap"]); $server->addFunction("add"); $server->handle();

That would expose add() as a SOAP function.

But:

  • Requires enabling the SOAP extension in PHP.

  • Very “2000s enterprise-y”, not modern REST JSON style.

🔹 REST in PHP

  • PHP doesn’t have a native RestServer .

  • But REST is just HTTP + JSON , so you can do it manually:

if ($_SERVER['REQUEST_METHOD'] === 'GET' && $_GET['action'] === 'sum') { $a = intval($_GET['a']); $b = intval($_GET['b']); echo json_encode(['result' => $a + $b]); }

This returns JSON directly.
Modern PHP frameworks (Laravel, Symfony, Slim, Lumen) give you routing, controllers, and JSON serialization out-of-the-box.

🔹 Moodle’s case

  • Moodle did not rely on PHP’s SoapServer or an external REST framework.

  • Instead, Moodle wrote its own dispatcher layer in PHP (inside /webservice/ ).

  • That layer:

    • Parses incoming params.

    • Looks up the corresponding Moodle PHP function (e.g., core_user_get_users ).

    • Executes it.

    • Serializes the result (JSON, XML, SOAP response).

So Moodle’s framework is essentially a mini version of what Laravel or Symfony would give you , but built in 2010s style, hand-rolled, to fit Moodle’s architecture.

✅ So to answer directly:

  • PHP doesn’t auto-expose functions as SOAP/REST web services.

  • For SOAP → PHP has SoapServer .

  • For REST → you build your own dispatcher (or use a framework).

  • Moodle → built its own dispatcher in PHP, not using external frameworks.

Perfect 👍 — here are the main pitfalls groups run into when evolving an old PHP site into a mobile+API-based architecture:

🔹 1. UI/Logic Separation Is Messy

  • In old PHP apps, business logic is often tangled with HTML .
    Example:

    while ($row = $db->fetch()) { echo "<tr><td>{$row['name']}</td></tr>"; }

  • Pulling this out into a clean getUsers() function can require major refactoring.

  • Teams underestimate how time-consuming untangling PHP spaghetti is.

👉 Fix : Start with the most important features (e.g., “courses list”), refactor step by step. Don’t try to API-ify everything at once.

🔹 2. Performance Issues

  • Old PHP apps often do DB queries directly in page scripts → fine for one user, but heavy API usage can overload the DB.

  • Example: Mobile app calls /api/courses 50 times per minute across 1,000 students → DB melts.

👉 Fix :

  • Add caching (Redis, Memcached).

  • Batch queries (send all needed data in one API response).

  • Rate-limit APIs.

🔹 3. Authentication & Security

  • Exposing raw PHP functions is dangerous:

    • Without tokens, anyone could call /api/deleteUser?id=42 .

    • Session cookies may not work well with mobile clients.

👉 Fix :

  • Use API tokens , JWT , or OAuth2.

  • Separate mobile app authentication from web session cookies.

  • Never expose DB IDs blindly — check permissions server-side.

🔹 4. Consistency Between Old UI and API

  • Old PHP UI might have business rules (e.g., “don’t show hidden courses”), but the new API may skip those checks.

  • Result: mobile app shows different data than website.

👉 Fix :

  • APIs should call the same business logic functions used by the PHP UI (like Moodle does).

  • Don’t duplicate rules in two places — centralize them in library functions.

🔹 5. Versioning & Backward Compatibility

  • Once mobile apps are published, you can’t instantly update all clients.

  • If API changes break old apps, users get errors.

👉 Fix :

  • Version your APIs ( /api/v1/courses , /api/v2/courses ).

  • Keep old versions alive until apps are updated.

🔹 6. Developer Skills Gap

  • PHP-only devs may not know React Native.

  • React devs may not know PHP.

  • Coordination can slow projects down.

👉 Fix :

  • Define a clear API contract (OpenAPI/Swagger docs).

  • Backend and frontend teams can then work in parallel.

✅ Summary

The big pitfalls are:

  1. Tangled legacy code → hard to extract APIs.

  2. API load/performance issues.

  3. Security & authentication.

  4. Inconsistent rules between UI and API.

  5. API versioning headaches.

  6. Skill gap between PHP and mobile devs.

But if done carefully (like Moodle did), the payoff is huge → mobile apps + future-proof API architecture.

ChatGPT said:

👏 Excellent catch — you’re right to compare the two. Let’s tease it apart carefully:

🔹 Old PHP website (SSR model)

  • Each student visiting /courses.php →

    • PHP runs.

    • Queries DB.

    • Builds HTML.

    • Sends it.

So if 1,000 students load the page in the same minute → you get ~1,000 DB queries.
👉 Yes, the DB can “melt” if queries are heavy or not cached.

But… in practice:

  • Students don’t usually refresh constantly.

  • HTML pages are “heavier” but accessed less frequently.

  • Web admins can enable page caching (e.g., Moodle has “Course cache” and “Theme cache”).

  • HTTP reverse proxies (Varnish, Nginx) can cache the rendered HTML for non-logged-in pages.

🔹 New API-based model

  • Mobile apps tend to poll more frequently .

  • Example:

    • Every time a student navigates in the app → new API call.

    • Some apps refresh every 30 seconds (“check for new messages, new grades, etc.”).

  • So while the website might hit /courses.php once per session, the mobile app might hit /api/courses dozens of times per session.

So same 1,000 students , but instead of 1,000 calls/minute, you can suddenly have 50,000 calls/minute.
👉 That’s why DB stress shows up much faster with APIs.

🔹 Key difference

  • SSR web app → fewer, heavier requests (HTML + CSS).

  • API mobile app → many more, smaller requests (JSON), but potentially orders of magnitude more frequent.

🔹 How real-world systems solve it

  • Use caching (Redis/Memcached, Moodle has MUC cache).

  • Use batching (send all data in one JSON response instead of 10 calls).

  • Use push notifications (instead of constant polling).

  • Use rate limiting (protects backend if clients misbehave).

✅ So yes, old PHP websites can also overload the DB under high load, but API-based mobile apps amplify the load pattern because they encourage more frequent, granular calls.

  • Moodle core → written in PHP since 2002.

  • First Moodle Mobile app → around 2011–2013.

  1. Write Once, Run Anywhere

    • Moodle wanted one codebase for iOS + Android.

    • Cordova provided that, wrapping a web app in a WebView.

    • Angular made it easier to structure a large client-side app (compared to jQuery).

  2. Developer familiarity

    • Moodle already had lots of web developers (PHP + JS), but not native mobile devs (Java/Kotlin for Android, Obj-C/Swift for iOS).

      • iOS app

      • Android app

      • Potentially a web app too (though in practice, the Moodle Mobile app was positioned mainly for mobile).

🔹 Why not native apps?

  • Building two separate native apps (Java for Android + Obj-C/Swift for iOS) was:

    • Expensive.

    • Hard to maintain.

    • Out of alignment with Moodle’s open-source, community-driven model (contributors were mainly web devs, not native mobile devs).

So native was never really an option.

🔹 Why not React Native?

  • React Native simply didn’t exist yet when Moodle Mobile began.

    • A development team skilled in Angular.

    • Hundreds of institutions relying on it.

Switching would’ve required a full rewrite , so they stuck with Angular.

🔹 Downsides of Angular + Cordova (acknowledged later)

  • Performance : Running in WebView, not native UI, so slower compared to React Native or Flutter.

  • Look & feel : Feels “webby”, not 100% native.

  • Maintenance : AngularJS eventually became outdated, requiring migration to newer Angular.

Section 7. Performance comparison between PHP backend and Node Express backend

You said:
Would Moodle PHP server acting as REST API backend be having performance issues as compared to later technology Node Express server?
ChatGPT said:

Excellent question 👍 — this gets to the heart of why people sometimes worry about PHP backends vs Node.js backends . Let’s break it down in the Moodle (PHP) vs Node/Express (JS) context.

🔹 How PHP (Moodle) REST backend works

  • Moodle = classic Apache + PHP + MySQL/Postgres stack.

  • Each API request (e.g., /webservice/rest/server.php?wsfunction=core_course_get_courses ) →

    1. Apache spawns a PHP worker.

    2. PHP initializes Moodle bootstrap code (quite heavy).

    3. Executes the requested function (DB queries, business logic).

    4. Serializes output as JSON.

    5. Ends process.

👉 Every request is stateless . Each call re-loads Moodle core libraries and re-connects to DB.

🔹 How Node/Express REST backend works

  • Node.js uses a long-running event loop.

  • When server starts:

    • Express app initializes once.

    • Routes and middleware stay in memory.

  • Each incoming request is handled by the event loop → non-blocking IO → same Node process continues running.

👉 Much less overhead per request, and reusing in-memory objects/cache is easy.

🔹 Performance comparison

  1. Startup overhead per request

    • PHP (Moodle) : Higher → bootstrap on every call.

    • Node.js : Lower → app already initialized.

  2. Concurrency model

    • PHP : Each request handled by separate process/thread (depends on Apache/FPM). Scales OK, but higher memory cost.

    • Node.js : Single-threaded event loop with async IO → handles many requests with lower memory.

  3. Throughput

    • Benchmarks usually show Node.js can handle more requests/sec than raw PHP under load, especially for API-heavy apps.

    • BUT — for apps that are DB-bound (lots of SQL queries, joins, etc.), the database becomes the bottleneck, not PHP vs Node.

  4. Caching

    • Both can use Redis/Memcached to reduce DB load.

    • Node.js makes in-process caching easy, while PHP needs external caching since each request is fresh.

🔹 Why Moodle stayed with PHP backend

  • Moodle is not a “microservice” system → it’s a huge monolithic LMS.

  • Rewriting in Node.js would be massive (decade-long effort).

  • Most Moodle performance issues come from database queries and scaling architecture , not just PHP runtime.

  • With OPcache, Redis caching, HTTP caching , Moodle can scale to millions of users.

🔹 Practical takeaway

  • Yes , a Moodle REST API backend in PHP has more per-request overhead than a modern Node/Express backend.

  • But , in real-world LMS usage:

    • DB is the bottleneck.

    • Caching solves most of the PHP statelessness inefficiency.

    • Universities often run Moodle behind load balancers with horizontal scaling.

So Moodle’s PHP backend is “slower” in theory, but not a blocker in practice , given proper infra.

✅ Answer in short :

  • Moodle’s PHP REST backend does have more overhead per request than Node/Express.

  • But for real-world workloads, DB and caching dominate performance — so with tuning, Moodle scales fine.

  • Node.js would win on raw efficiency, but not enough to justify rewriting Moodle.

2020s

  • SPA vs SSR :
